1 # Localize HTML Tidy {#localize}
2 
3 Thank you for your interest in helping us localize HTML Tidy and LibTidy. Users
4 throughout the world will thank you.
5 
6 The document describes Tidy's localization philosophy and instructs you on how
7 you can use standard `gettext` tools to generate language and region
8 localizations that will work with Tidy. Optionally instructions are included in
9 the event that you want to build Tidy with your new language.
10 
11 ## Introduction
12 
13 HTML Tidy is built around the localization file `language_en.h`; without this
14 file HTML Tidy will not work. As such _all_ language localization work
15 originates from this single file.
16 
17 Language localizations use header files that are identical to `language_en.h`,
18 except that they have different strings. For the convenience of language
19 translators, though, Tidy source code includes a Ruby `poconvert.rb` script
20 that enables _optional_ gettext PO/POT work streams that may be more comfortable
21 to them.
22 
23 
24 ### PO and POT files
25 HTML Tidy provides PO and POT files for language translations. The file
26 `tidy.pot` is the correct template to use as a basis for new translations. In a
27 typical `gettext` workflow a translator will use the `tidy.pot` file to create a
28 language translation PO file that contains original English strings and the
29 translated strings.
30 
31 If a language has already been translated (or if the translation has begun) then
32 PO files may already exist. These files are named `language_ll.po` or
33 `langage_ll_CC.po`, where `ll` represents the language code, and optionally,
34 `CC` represents the region code of the translation.
35 
36 Tidy does not use MO files that `gettext` tools generate from PO files.
37 
38 Please note that these PO and POT files are provided for translator convenience
39 only. Tidy's [header files](#h-files) constitute the true, controlled source
40 code for Tidy.
41 
42 
43 ### H files
44 
45 Tidy does not use `gettext` to display strings and so `gettext`-generated MO
46 files are not necessary. Instead translated PO files must be converted to Tidy's
47 language header H file format. Translators are not required to perform this
48 step, but we provide a tool to perform this function if desired.
49 
50 
51 ### Differences for translators
52 
53 Experienced users and translators of PO files may note that we use the PO file's
54 `msgctxt` field a bit uniquely. Rather than point to a line in the source code,
55 it contains a reference to the string's identifier. Because the PO format does
56 not allow for arbitrary metadata this is a requirement for generating our
57 header files.
58 
59 If you're the type of translator the does dig into the source code, then this
60 `msgtext` symbol is still useful to you and adds a single extra step to finding
61 where a string is in context: a symbol or string search using the `msgctxt`
62 value will reveal the context in source code.
63 
64 Finally the `msgid` field is a throwaway; Tidy's language tools do not use this
65 value and so it's only for the translator's convenience. This fact makes it
66 convenient for translators to translate from languages other than English,
67 which is fully supported by our tools.
68 
69 
70 ### `poconvert.rb` versus `gettext`' tools
71 
72 Please don't use `gettext`' tools with our PO and POT files (unless you are
73 using our strings for a different project). Instead all workflows can be
74 accomplished with our `poconvert.rb` tool.
75 
76 [More information about this tool](#h-files) can be found below.

79 ## How to Contribute
80 
81 ### Find or Create the Translation Files
82 If you've not already cloned the HTML Tidy source code repository that will be
83 your first step.
84 
85 In the `localize\translations\` directory you can find existing languages, e.g.,
86 
87  - `tidy.pot` (Tidy's POT template for translations).
88  - `language_en_gb.po` (British English variants for the built in language)
89  - …and perhaps more.
90 
91 In the `src\` directory you can find the master files for existing languages,
92 e.g.,
93 
94  - `language_en.h` (Tidy's native, built-in language, mostly U.S. English)
95  - `language_en_gb.po` (British English variants for the built in language)
96  - …and perhaps more.
97 
98 Although the header files are the master files for HTML Tidy, we understand that
99 not all potential translators want to edit C files directly. Therefore as an
100 option, the following workflow to use POT and PO files is offered.
101 
102 If the language that you want to work on is already present:
103 
104  - Simply open the file in your favorite PO editor and then get to work.
105  - Note that although you can use a text editor, we recommend that you use a
106  dedicated PO editor so that you don't accidentally make the file illegible
107  to our conversion utility.
108 
109 If the language that you want to work on is _not_ already present:
110 
111  - You can open `tidy.pot` in your favorite PO editor and use its functions
112  to begin a new translation into your desired language.
113  - Note that although you can use a text editor, we recommend that you use a
114  dedicated PO editor so that you don't accidentally make the file illegible
115  to our conversion utility.
116  - To perform the work manually:
117  - Copy `tidy.pot` to `language_ll.po` (for a non-regional variant, or base
118  language), or to `language_ll_cc.po` (for a region-specific variant),
119  where `ll` indicates the two letter language code and `cc` indicates the
120  two letter region or country code.
121  - Change the pertinent PO header section accordingly.
122  - Use `poconvert.rb` to generate a PO:
123  - `poconvert.rb msginit --locale ll`, where `ll` indicates the language
124  code for the language you want to translate to. The tool recognizes the
125  same languages as `gettext`' `msginit`. If your chosen language is not
126  supported, then please see the manual method, above.
127  - See also `poconvert.rb help` for more options.
128  - See GNU's [The Format of PO Files](https://www.gnu.org/software/gettext/manual/html_node/PO-Files.html)
129  for more specific instructions and important information.
130 
131 ### Issue a Pull Request to HTML Tidy
132 
133 Once your translation is complete commit your entire HTML Tidy repository to
134 GitHub and issue a pull request (PR) against the `master` branch. If accepted a
135 friendly developer will convert your PO into a format useful to Tidy if your
136 PR is a PO, or will simply merge your changed header file if you changed it
137 directly.
138 
139 You are also welcome to perform any conversions yourself, add new languages to
140 Tidy, and issue a PR for the whole change.
141 
142 
143 ### Using Git appropriately
144 
145  1. Fork the repository to your GitHub account.
146  2. Optionally create a **topical branch** - a branch whose name is succinct but
147  explains what you're doing, such as "localize Portuguese".
148  3. Make your changes, committing at logical breaks.
149  4. Push your work to your personal account.
150  5. [Create a pull request](https://help.github.com/articles/using-pull-requests).
151  6. Watch for comments or acceptance.

154 ### Repository Notes
155 
156 If you are working with PO files then please **only** commit PO files with
157 _English_ `msgid` fields. The `gettext` convention specifies only English
158 `msgid`, and other translators may not understand the original strings.
159 
160 Our `poconvert.rb` script can generate PO files using another language as
161 `msgid`. This can be very useful if it's easier for you to translate from
162 another language instead of English. It can also be useful for translating from
163 a base language to a regional variant, such as from Spanish to Mexican Spanish.
164 
165 If you choose to work locally with a non-English PO, you can easily convert
166 your PO to a Tidy header file and back to an English-based PO using our
167 `poconvert.rb` script. See its documentation (`poconvert.rb help`) for
168 instructions.

171 ## Adding Languages to Tidy
172 
173 Although we don't require you to follow these steps to contribute a language
174 to Tidy, you may want to add the language to Tidy yourself to test the
175 translation, or to save one of the developer team a few extra steps.
176 
177  - Generate the header files if necessary:
178  - Convert your PO file to a Tidy header file by executing
179  `poconvert.rb msgfmt <path_to_your_file.po>`. Note that on Windows you
180  will likely have to preface this line with `ruby`.
181  - The tool should generate a file named `language_ll_cc.h` in the working
182  directory, where `ll_cc` will be replaced with the language/region of your
183  translation.
184  - Copy this `.h` file into `src\`.
185  - Modify Tidy's source:
186  - Edit the file `src\language.c` to ensure that the new `.h` file you added
187  is in the header include section.
188  - Look for the `static tidyLanguagesType tidyLanguages` structure starting
189  near line 40, and look for the comment `These languages are installed.`.
190  You can add your new language to the list along with the other languages
191  present, following the same format.
192  - Build Tidy:
193  - Build Tidy per the usual instructions, and try it out using the `-lang`
194  option.

197 ## Best Practices
198 
199 ### Language Inheritance
200 
201 HTML Tidy will fallback from the specified language to the base language and
202 then finally to the default English as required. This means, for example, a
203 programmer might set `libtidy` to use “es_mx”, and if it doesn’t exist Tidy
204 will automatically use “es”. If that doesn’t exist `libtidy` will continue to
205 use whatever language it is currently using.
206 
207 
208 ### String Inheritance
209 
210 HTML Tidy will also fallback for individual strings. For example if `libtidy`
211 is set to use “es_mx” and a particular string is requested and not found, the
212 library will look for the string in “es”. If the string is not found there then
213 the “en” string will be given.
214 
215 
216 ### Base Language First and Regional Variants
217 
218 Because of this inheritance we hope to localize base languages first, as the
219 only strings then required for regional variants are the strings that differ.
220 This will help us keep HTML Tidy and `libtidy` small.
221 
222 If you are working on a regional variation (such as “us_CA”) please only
223 localize strings that are actually _different_ from the base language!
224 
225 
226 ### Positional Parameters
227 
228 Please note that HTML Tidy does not currently support positional parameters. Due
229 to the nature of most of Tidy's output, it's not expected that they will be
230 required. In any case, please translate strings so that substitution values are
231 in the same order as the original string.

239 ### Command line option
240 
241 Use the `-lang`/`-language` option and specify a POSIX or Windows language name.
242 This option should be first option used because the console application parses
243 and acts on options first-in, first-out.
244 
245 ### Changing your locale
246 
247 On Unix/Mac and Linux operating systems you can change your shell’s locale
248 temporarily with:
249 
250 `export LANG=en_GB`
251 `export LC_ALL=en_GB`
252 
253 …substituting, of course the language of your choice.
254 
255 ### East Asian Languages
256 
257 East Asian languages are completely supported and have been tested on Linux,
258 Mac OS X, and Windows, although Windows requires you to set your operating
259 system (not the console locale!) to an East Asian locale to enable this in
260 Windows Console and PowerShell. Note that PowerShell ISE always supports East
261 Asian languages without requiring you to change your operating system locale.

264 ## gettext
265 
266 Although HTML Tidy uses `gettext`-compatible tools and PO files for language
267 localization, Tidy itself does _not_ use `gettext`. Tidy's build philosophy is
268 build it anywhere and build it with anything. As `gettext` is not universally
269 available on every platform under the sun, Tidy cannot count on `gettext`.
270 
271 Instead Tidy builds all translations into its library (and command line
272 executable if built monolithically), and can run on virtually any general
273 purpose computer with any operating system.
274 
275 While this does not pose a significant problem for storage or execution space
276 on modern PC's, we understand that certain applications may still be space
277 critical. As such it's trivial to build Tidy without this extra language
278 support using the `-DSUPPORT_LOCALIZATIONS=NO` switch.

281 ## poconvert.rb
282 
283 Tidy's source code includes a Ruby batch file called `poconvert.rb` which can be
284 used to generate POT, PO, and H files, and convert them back and forth. It has
285 been designed to work in a similar fashion as `gettext`'s tools, and includes
286 conveniences that let translators work in different source languages. Please
287 use `poconvert.rb help` for complete information (`ruby poconvert.rb help` on
288 Windows).
289 
290 Note that you must install Ruby on your system, as well as install the required
291 dependencies. These can be manually installed with `[sudo] gem install xxx`,
292 where `xxx` represents the packages listed in `Gemfile`. For convenience, if you
293 have the Bundler gem installed, you can `bundle install` for automated
294 dependency installation.
295 
296 Also take note of these two important characteristics:
297 
298 - `poconvert.rb` is currently dependent on its current path. You can move it
299  from its current location, but you will have to change the values of the
300  `@@default_en` and `@@header_template` variables within the script.
301 - All files will be output in the current working directory. This will prevent
302  accidental overwrites of important files while we all get used to the
303  workflows.
304 
305 Below are some sample workflows.
306 
307 
308 ### Create a new POT file
309 
310 Although we provide `tidy.pot` in the source, you can generate your own.
311 
312 `./poconvert.rb xgettext`
313 
314 This will put a fresh, new copy of `tidy.pot` in the working directory.
315 
316 
317 ### Create a new POT file with non-English `msgid` strings
318 
319 Although `gettext` officially recognizes English as the one, true source
320 language for PO and POT files, if you're more comfortable translating from a
321 non-English language, we can support you.
322 
323 `./poconvert.rb xgettext <language_cc_ll.h>`
324 
325 Where `<language_cc_ll.h>` is the path to an existing Tidy language header file.
326 This will produce a `tidy.pot` using the translated strings as `msgid`, using
327 English as a backup when translated strings are not present.
328 
329 This can be valuable in producing regional variant translations, e.g., when
330 translating from `es` to `es_mx`.
331 
332 
333 ### Convert an existing H to PO
334 
335 In many cases you may want to have a fresh PO generated from a Tidy H file.
336 This can be accomplished with:
337 
338 `./poconvert.rb msgunfmt <language_cc_ll.h>`
339 
340 
341 ### Convert an existing H to PO using a different `msgid` language
342 
343 If you want to generate a fresh PO file from a Tidy H file, but _also_ want to
344 have untranslated strings from a language other than English, try:
345 
346 `./poconvert.rb msgunfmt <language_cc_ll.h> --baselang=<other-language_cc_ll.h>`
347 
348 
349 ### Create a blank PO file for a particular region
350 
351 `./poconvert.rb msginit`
352 or
353 `./poconvert.rb msginit --locale=LOCALE`
354 
355 The first example will try to guess your current region, and the second will
356 use a region specified.
357 
358 Tidy only knows about the same regions that `gettext` knows; if our `msginit`
359 does not recognize the region you specify, you will have to create a new PO
360 and modify the region settings yourself.
361 
362 To create the blank PO using `msgid` strings from a different Tidy language,
363 you can use:
364 
365 `./poconvert.rb msginit <language_cc_ll.h> [--locale=LOCALE]`
366 
367 
368 ### Create a Tidy Language Header H file
369 
370 When you're ready to include the language in Tidy, you can generate its header
371 file with:
372 
373 `./poconvert.rb msgfmt <language_cc_ll.po>`
374 
375 In the event you are creating a regional variant of a language, it's an
376 excellent idea to have Tidy exclude strings that are already present in the
377 parent language in order to reduce library and executable size. For example
378 if `es` already includes the string "archivo" there is no reason for your
379 translation to `es_mx` to include it, too. You can tell `poconvert.rb` to
380 exclude strings matching another localization like so:
381 
382 `./poconvert.rb msgfmt <language_cc_ll.po> --baselang=<other-language_cc_ll.h>`
383 
384 
385 ### Prepare your non-English PO for a PR
386 
387 Although we have provided tools to allow you to work in languages other than
388 English, we can only accept PO's in the repository that have English `msgid`
389 fields. It's easy to convert your PO back to English:
390 
391 `./poconvert msgfmt <language_cc_ll.po>`
392 
393 `./poconvert msgunfmt <language_cc_ll.h>`
394 
395 The first command converts your non-standard PO into a Tidy Language Header
396 file, and the second will create a fresh, new PO file from the header that
397 you've just created.
398 
399 
400 ### Update your PO to match the new POT
401 
402 If Tidy's POT changes, e.g., new strings are added, new comments, etc., the
403 simplest way to update your PO is to convert it to a header (which normalizes
404 it to the latest Tidy standard), and then convert the header to a new PO again.
405 
406 `./poconvert msgfmt <language_cc_ll.po>`
407 
408 `./poconvert msgunfmt <language_cc_ll.h>`
