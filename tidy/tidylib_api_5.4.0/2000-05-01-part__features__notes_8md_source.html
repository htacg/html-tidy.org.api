3 ## Indenting output for readability
4 
5 Indenting the source markup of an HTML document makes the markup easier
6 to read. **Tidy** can indent the markup for an HTML document while recognizing
7 elements whose contents should not be indented. In the example below, **Tidy**
8 indents the output while preserving the formatting of the `<pre>` element:
9 
10 Input:
11 
12 ~~~
13 <html>
14  <head>
15  <title>Test document</title>
16  </head>
17  <body>
18  <p>This example shows how Tidy can indent output while preserving
19  formatting of particular elements.</p>
20 
21  <pre>This is
22  <em>genuine
23  preformatted</em>
24  text
25  </pre>
26  </body>
27  </html>
28 ~~~
29 
30 Output:
31 
32 ~~~
33 <html>
34  <head>
35  <title>Test document</title>
36  </head>
37 
38  <body>
39  <p>This example shows how Tidy can indent output while preserving
40  formatting of particular elements.</p>
41 <pre>
42 This is
43 <em>genuine
44  preformatted</em>
45  text
46 </pre>
47  </body>
48 </html>
49 ~~~
50 
51 **Tidy**’s indenting behavior is not perfect and can sometimes cause your
52 output to be rendered by browsers in a different way than the input.
53 You can avoid unexpected indenting-related rendering problems by setting
54 `indent:no` or `indent:auto` in a config file.

57 ## Preserving original indenting not possible
58 
59 **Tidy** is not capable of preserving the original indenting of the markup
60 from the input it receives. That’s because **Tidy** starts by building a clean
61 parse tree from the input, and that parse tree doesn’t contain any
62 information about the original indenting. **Tidy** then pretty-prints the parse
63 tree using the current config settings. Trying to preserve the original
64 indenting from the input would interact badly with the repair operations
65 needed to build a clean parse tree, and would considerably complicate the
66 code.

69 ## Encodings and character references
70 
71 **Tidy** defaults to assuming you want output to be encoded in UTF-8.
72 But **Tidy** offers you a choice of other character encodings: US ASCII, ISO
73 Latin-1, and the ISO 2022 family of 7 bit encodings.
74 
75 **Tidy** doesn't yet recognize the use of the HTML `<meta>` element for
76 specifying the character encoding.
77 
78 The full set of HTML character references are defined. Cleaned-up output
79 uses named character references for characters when appropriate. Otherwise,
80 characters outside the normal range are output as numeric character
81 references.

91 ## Cleaning up presentational markup
92 
93 Some tools generate HTML with presentational elements such as `<font>`,
94 `<nobr>`, and `<center>`. **Tidy**'s `‑clean` option will replace those elements
95 with `<style>` elements and CSS.
96 
97 Some HTML documents rely on the presentational effects of `<p>` start
98 tags that are not followed by any content. **Tidy** deletes such `<p>` tags
99 (as well as any headings that don’t have content). So do not use `<p>`
100 tags simply for adding vertical whitespace; instead use CSS, or the
101 `<br>` element. However, note that **Tidy** won’t discard `<p>` tags that
102 are followed by any non-breaking space (that is, the `&nbsp;` named
103 character reference).

106 ## Teaching Tidy about new tags
107 
108 You can teach **Tidy** about new tags by declaring them in the
109 configuration file, the syntax is:
110 
111 ~~~
112 new-inline-tags: tag1, tag2, tag3
113 new-empty-tags: tag1, tag2, tag3
114 new-blocklevel-tags: tag1, tag2, tag3
115 new-pre-tags: tag1, tag2, tag3
116 ~~~
117 
118 The same tag can be defined as **empty** and as **inline**, or as **empty**
119 and as **block**.
120 
121 These declarations can be combined to define a new empty
122 inline or empty block element, but you are not advised to declare
123 tags as being both **inline** and **block**.
124 
125 Note that the new tags can only appear where **Tidy** expects inline
126 or block-level tags respectively. That means you can’t place
127 new tags within the document head or other contexts with restricted
128 content models.

131 ## Ignoring PHP, ASP, and JSTE instructions
132 
133 **Tidy** will gracefully ignore many cases of PHP, ASP, and JSTE
134 instructions within element content and as replacements for attributes,
135 and preserve them as-is in output; for example:
136 
137 ~~~
138 <option <% if rsSchool.Fields("ID").Value
139  = session("sessSchoolID")
140  then Response.Write("selected") %>
141  value='<%=rsSchool.Fields("ID").Value%>'>
142  <%=rsSchool.Fields("Name").Value%>
143  (<%=rsSchool.Fields("ID").Value%>)
144 </option>
145 ~~~
146 
147 But note that **Tidy** may report missing attributes when those are “hidden”
148 within the PHP, ASP, or JSTE code. If you use PHP, ASP, or JSTE code to
149 create a start tag, but place the end tag explicitly in the HTML markup,
150 **Tidy** won’t be able to match them up, and will delete the end tag. In that
151 case you are advised to make the start tag explicit and to use PHP, ASP, or
152 JSTE code for just the attributes; for example:
153 
154  `<a href="<%=random.site()%>">do you feel lucky?</a>`
155 
156 **Tidy** can also get things wrong if the PHP, ASP, or JSTE code includes
157 quotation marks; for example:
158 
159 `value="<%=rsSchool.Fields("ID").Value%>"`
160 
161 **Tidy** will see the quotation mark preceding `ID` as ending the
162 attribute value, and proceed to complain about what follows.
163 
164 **Tidy** allows you to control whether line wrapping on spaces within PHP, ASP,
165 and JSTE instructions is enabled; see the `wrap-php`, `wrap-asp`,
166 and `wrap-jste` config options.

169 ## Correcting well-formedness errors in XML markup
170 
171 **Tidy** can help you to correct well-formedness errors in XML markup. **Tidy**
172 doesn't yet recognize all XML features, though; for example, it doesn't
173 understand CDATA sections or DTD subsets.
