1 # Contributing to HTML Tidy {#contributing}
2 
3 So you want to contribute to Tidy? Fantastic! Here's a brief overview on how best to do so.
4 
5 ### Support request
6 
7 If you are having trouble running console `Tidy`, or using the `LibTidy` API in your own project, then maybe the best places to get help is either via a comment in [Tidy Issues](https://github.com/htacg/tidy-html5/issues), or on the [Tidy Mail Archive](https://lists.w3.org/Archives/Public/html-tidy/) list.
8 
9 In either place please start with a short subject to describe the issue. If it involves running Tidy on an html file, or if it’s an API question, make sure to include:
10 
11  - the version: `$ tidy -v`
12  - what was the configuration used
13  - a small sample input
14  - the output
15  - the _expected_ output expected
16  - some sample code (if an API question).
17 
18 These data will make replication of your issue much simpler for us.
19 
20 If you do add sample HTML input, then it can also be very helpful if that sample **passes** the W3C [validator](https://validator.w3.org/#validate_by_upload). Tidy attempts to follow all current W3C standards.
21 
22 If you are able to build tidy from [source](https://github.com/htacg/tidy-html5) (requires [CMake](https://cmake.org/download/)), and you can find the problem in the source code, then read on about how you can create a Pull Request (“PR”) to share your code and ideas.
23 
24 
25 ### What to change
26 
27 Here are some examples of things you might want to make a PR for:
28 
29  - New features
30  - Bug fixes
31  - Inefficient blocks of code
32  - Memory problems
33  - Language translations
34 
35 If you have a more deeply-rooted problem with how the program is built or some of the stylistic decisions made in the code, it is best to [create an issue](https://github.com/htacg/tidy-html5/issues/new) before putting the effort into a pull request. The same goes for new features - it might be best to check the project's direction, existing pull requests, and currently open and closed issues first.
36 
37 Concerning the “Tidy Code Style,” checkout [CODESTYLE.md](CODESTYLE.md), but looking at existing code is the best way to get a good feel for the patterns we use.
38 
39 
40 ### Using Git appropriately
41 
42  1. Fork the repository to your GitHub account.
43  2. Optionally create a **topical branch**, a branch whose name is succint but explains what you're doing, such as "feature/add-new-lines".
44  3. Make your changes, committing at logical breaks.
45  4. Push your work to your personal account.
46  5. [Create a pull request](https://help.github.com/articles/using-pull-requests).
47  6. Watch for comments or acceptance.
48 
49 Please note - if you want to change multiple things that don't depend on each
50 other, it is better to use `branches`, and make sure you check the master branch back out before making more changes - that way we can take in each change seperately, otherwise Github has a tendancy to combine your requests into one.
51 
52 If you are a continuing contributor then you will need to `rebase` your fork, to htacg `next`, **before** doing any more work, and likewise branches, otherwise we may not be able to cleanly merge your PR. This is a simple process:
53 
54 ```
55 $ git remote add upstream git@github.com:htacg/tidy-html5.git # once only
56 $ git checkout next
57 $ git status
58 $ git stash # if not clean
59 $ git fetch upstream
60 $ git rebase upstream/next
61 $ git stash pop # if required, and fix conflicts
62 $ git push # update the fork next
63 ```
64 
65 This can be repeated for other branches, too.
66 
67 ### Help Tidy Get Better
68 
69 It goes without saying **all help is appreciated**. We need to work together to make Tidy better!
